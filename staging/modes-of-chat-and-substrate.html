Modes of Chat and Substrate

By Cliff Rosen, Founder & CTO, Orchestrator Studios

April 2026

Apps before chat

Before chat became a productive capability, the structure of an application was settled territory. An app, in its general form, decomposes into three things:

  • The user — the person doing the work.
  • The substrate — the structured persistent state the app reads from and writes to. A CRM has records. A word processor has documents. A spreadsheet has cells. A file system has files and directories. The substrate is what gives the app its memory and its shape.
  • The UI — the visual interface for interacting with the substrate. Buttons, forms, lists, dashboards.

The user's overall experience of working with the app — the UX — is composed of the UI plus how the user navigates through it. In a pre-chat app, the UX is essentially the UI in motion. Once chat is added (in any of the modes below), chat becomes part of the UX too: chat isn't a layer separate from UX but a component of it, alongside the UI.

Not every part of an app is everyday-use. Many apps include fabricator portions — places where the substrate's structure itself gets shaped, distinct from the everyday-use portions where the user operates within an existing structure. Salesforce has Setup, where admins create custom objects, define fields, configure validation rules. Word has its template-and-style system. Excel has macros and named ranges. The fabricator portion and the everyday-use portion are often separate experiences — sometimes for different roles (admin vs. end user), sometimes just for different mental modes.

These patterns were established decades before chat. The architecture of substrate + UI + user, with a fabricator/use distinction layered on top, is more or less how every productive piece of software was built and operated.

Then chat showed up

Three or four years ago, a new capability arrived in productive form: chat. Specifically, a user sending messages to a large language model running in a loop with tools — an LLM that can call tools, see their results, and continue reasoning across multiple turns within a single user prompt.

Even before LLMs made chat truly viable, the app ecosystem had been trying to incorporate chat-like interfaces, typically as support sidekicks. A help bot tucked into a corner. A question-answerer next to the real product. The pattern was known, even if the experience wasn't great. After ChatGPT in late 2022, this remained the obvious first move: bolt an actual capable chat onto a traditional app and let it explain the UI, answer questions, walk users through tasks. Many products shipped exactly this. A fine starting point — but a small one.

Where chat could go beyond the sidekick

What can chat do for an app beyond the sidekick role? The substrate is the point of reference — the thing being collaborated on. The variation across configurations is in how the UI and the chat divide up the work of moving things in and out of the substrate, both being parts of the user's overall experience.

The four basic modes

Four configurations cover most of the landscape:

  • Classic App. A traditional CRM (Salesforce, Hubspot, Pipedrive). Records — contacts, accounts, opportunities — managed through forms, lists, and dashboards. No chat at any layer. The app has a fabricator portion in Settings, separate from everyday use, where admins shape the substrate's structure ahead of time. This is what most productive software still looks like.
  • Assisted App. The same CRM with a chat sidepanel that answers questions about the UI — "how do I set a follow-up reminder?" The chat explains; it doesn't move state. Many products shipped exactly this when chat became a viable capability. Useful but small.
  • Chat-Driven App. Chat becomes the primary channel for driving state changes — and sometimes for searches or analysis that would otherwise require navigating complex UI. But the UI is still there, and it remains the primary interface for browsing and consuming the structured data: dashboards, lists, account views. In a CRM, you might say "log a 30-minute call with Acme yesterday, follow-up next Tuesday" instead of clicking through forms — but you'd still scan the pipeline and review accounts through the UI. Vibecoding tools (Lovable, Bolt, Replit Agent) are one form of this — chat as the primary interface for the work of building, after which chat exits. Claude Code is another — chat-driven authoring on a file system. Instances vary in how much the user can also work directly with the substrate: TableThat is tight (chat is the intended path); Claude Code is loose (you can edit files manually, run commands, work alongside chat).
  • Pure Chat. ChatGPT, Claude in their default form. No app, no substrate at all. You can ask about CRM-style work or ad-hoc structured projects, but you can't implement either — there's no place to store records, query state, or persist structured content. The "projects" feature in ChatGPT and Claude is a thin organizing layer over conversations, not a substrate.

And then there's the Living Workspace

There's a really interesting fifth case that's hardly showing up. Almost nothing is purpose-built for it as a product category. But the interesting thing is that we already have a tool that works for it — not by design, but because of a structural property of how Claude Code happens to be built.

Claude Code is technically a Chat-Driven App — chat-driven authoring. But because its substrate is so low-level (the file system), chat can fabricate higher-level substrates in real time: schemas, structured documents, custom views, bespoke interfaces for the work in front of you. That capacity — chat fabricating higher-level structure on top of a low-level substrate — turns out to be exactly what makes the Living Workspace pattern possible. Claude Code didn't set out to be a Living Workspace. It just happens to be usable as one.

What makes the Living Workspace distinctive: building, using, and enhancing happen in one continuous chat session. The chat that built the substrate is also the chat that operates within it and continues to shape it as the work matures. Pure Chat can't support this pattern — there's no substrate to fabricate on top of. The Living Workspace requires both a low-level substrate and the chat-driven authoring that does the fabricating.

What this looks like in practice

Take a small project: putting together a competitive-landscape analysis for next week's strategy meeting. The work has structure — sources to gather, claims to extract, themes to synthesize, a draft to refine. It has stages, works-in-progress at each step, and quality requirements (citations, internal consistency, narrative cohesion). And there's no off-the-shelf app for it. The traditional choice has been to do the work in a generic environment and hold the orchestration in your head, or to build a system — which is too expensive to justify for a one-week project.

In a Living Workspace, you sit down with the chat and develop the structure for the analysis as you start working. Through conversation: a folder for sources, a schema for extracted claims, a synthesis document with sections, a draft. The chat helps build the structure, then helps fill it, then helps refine. As needs evolve, you ask chat to extend the structure — a new way to organize sources by credibility, a flag for claims needing follow-up, a custom view that summarizes progress. The UI itself emerges and shifts through chat as the work calls for it.

Building, using, and enhancing aren't separate phases. They're a single continuous activity. The competitive analysis ships at the end of the week. The custom workspace built for it gets discarded or carried forward — but its primary purpose was getting this deliverable done well, with orchestration that fit the actual shape of the work.

What this enables

The conventional wisdom about custom apps is that they're expensive — you build them when the volume of work justifies the upfront cost. For one-off projects, the rule has always been: don't build a tool, just do the work in whatever generic environment is available.

The Living Workspace changes that calculus. When the chat that fabricates the app is also the chat doing the work — and the app emerges as part of the work itself — the cost of "building the app" folds into the cost of doing the work. Even one-off projects can have a custom-fit app, because the app is a side effect of the work, not a separate project.

And once you're doing meaningful work — including one-offs — you need orchestration anyway. There's no way around managing the process, the works-in-progress, the quality checks, the artifacts that need to exist at each stage. The traditional choice has been: do this informally if the project is small, build a system if it's big. The Living Workspace offers a third option — build a system right-sized for this specific work, while you're doing the work, in chat.

That's the punchline. Most of what's been built around chat-in-apps so far is some flavor of the four basic modes. The Living Workspace is the configuration that's still mostly a possibility waiting to be inhabited.
