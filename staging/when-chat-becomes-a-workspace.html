When Chat Becomes a Workspace

A pattern in Living Workspace mode: chat fabricates the application a knowledge journey needs, exactly when the journey is ready for it.

By Cliff Rosen, Founder & CTO, Orchestrator Studios

April 2026

In a previous article, we mapped six modes of chat and substrate — from classic apps with no chat, through pure chat tools with no underlying app, all the way to the Living Workspace mode where chat creates a substrate and stays alongside it indefinitely. Two corners of that map were noted as surprisingly underpopulated: the Chat-Driven App and the Living Workspace.

This piece works through a particular pattern that lives in the Living Workspace, and steel-mans why it deserves more attention than it's currently getting. The pattern is both natural — most knowledge work already wants to behave this way — and structurally non-trivial: it only works because of a specific contract built on top of the file system.

The pattern

A user begins a knowledge journey in pure chat. They want to write a paper, plan a trip, draft a proposal, build a spec. Pure chat is the right tool for this stage. The work is open-ended; the user doesn't yet know what shape it will take. Brainstorming, exploring, framing, putting first thoughts down — chat handles all of these well precisely because no structure is imposed.

But as the work matures, something shifts. Decisions get made. Stakes go in the ground. What was an open question becomes a defined effort with parts that need managing:

  • The work needs to be decomposed into stages.
  • State needs to be persisted at each stage — not as messages buried in a long thread, but as discrete artifacts you can return to.
  • The user needs to review where things are without scrolling back through the conversation.
  • The user needs to interact with the stored state — edit it, advance it, branch it.

Pure chat is poorly suited to all of this. The conversation is a medium, not a substrate. There's nothing to read from or write to with structure. The very property that made pure chat good early on — its shapelessness — becomes a limitation as the work demands shape.

The conventional move is to leave chat. The user opens a separate tool — a project tracker, a wiki, a spreadsheet, whatever fits the shape the work has taken on. The chat session ends or is abandoned. Continuity of the conversation is traded for the structure the work needs.

The pattern we want to steel-man is different: the chat doesn't end. It transforms.

The screen layout shifts. The chat pane, which had occupied the full width of the screen, contracts to a left panel. A right pane materializes — a custom-built application designed for this specific knowledge journey. The chat continues alongside the application. The user can still talk to the chat, ask it to manipulate state in the application, ask it to surface or transform content, ask it to extend the application as the work evolves. The session is unbroken. The work has acquired a structured home, but the conversation that produced that home is still right there, still active, still shaping things.

Why this is hard

The pattern is appealing. The execution is not obvious.

If you just ask a chat to "manage all of this for me, including the structure, the state, the views, the transitions," it doesn't work well. The LLM behind the chat can't reliably maintain complex state across many turns. It loses track. It forgets earlier decisions. It contradicts itself. State that should be canonical drifts as the conversation proceeds. Long sessions become a slow accumulation of inconsistencies.

This is a real limitation, and it's the same property that made chat good for early stages. Chat has no place to write things down except into more messages. There is no canonical store. Every previous decision is recoverable only by reading earlier in the thread, and the LLM's working memory of those earlier decisions is approximate at best.

For the chat-becomes-a-workspace pattern to be viable, something has to give the LLM a place to put state — and a contract about what kind of state goes where. Without that contract, the pattern collapses back into pure chat with all its limitations, just dressed up with extra UI.

The enabler: a schema-data-views contract on the file system

What makes the pattern work is a layered structure on top of a persistent file system. Three pieces, forming a contract:

Schema

A defined shape for the kinds of things the work involves. For a research paper: outline, source list, claims, evidence, draft sections. For a trip: destinations, dates, bookings, itinerary, budget. For a pipeline spec: overview, inputs, outputs, stages, gates. The schema names the parts and how they relate.

Data

Actual content conforming to the schema — the user's specific outline, source list, draft, itinerary, spec. The data lives on disk, in files, organized according to the schema. It persists across turns; it persists across sessions.

Views

Ways of seeing and interacting with the data. A view of the source list is different from a view of the draft. A view of the timeline is different from a view of individual deliverables. Views are the application's UX — the right pane that materializes alongside the chat.

These three together form a contract. The schema declares what exists. The data is what's been produced. The views are how the user (and the chat) work with it. With the contract in place, the LLM doesn't have to maintain everything in working memory. It writes to files. It reads from files. It manipulates a view by manipulating the underlying data. The state is persistent and queryable; the LLM is doing what it's actually good at — reasoning about content, generating, transforming — rather than what it's not good at, which is reliably maintaining a large structured state across a long conversation.

The file system is what makes this contract possible. A chat platform whose substrate is the file system (Claude Code, in current form) can host this kind of contract trivially. A pure chat platform without a substrate cannot. This is one of the things B2 — the Living Workspace — actually unlocks, when used well.

A worked example: writing the first spec

Here's how the pattern actually played out in one concrete case. The Claude Code plugin we built for writing pipeline specifications didn't start as a plugin. It emerged during a single session whose original goal was to produce one specific spec.

The journey began in chat. I knew the methodology I wanted to follow: a schema-driven approach with fairly rigorous rules about what a pipeline spec should contain — overview, defined inputs and outputs, sequenced stages, gates between them, validation criteria. The first thing the chat and I did was talk through it. Pure chat, working out what should and shouldn't be in a spec, what the right fields were, what shape the output should take.

Then the conversation began to crystallize. We developed a template — a concrete schema for what a spec looks like, written down as a file in the workspace. We added instructions on top of the template — rules for what the chat should enforce, what should be auto-populated, what should require explicit input from me. We discussed how the spec should be persisted: JSON for the structured parts, Markdown for the narrative parts, organized in folders that reflected the spec's stages.

By this point, what had started as a chat about methodology had become a working skill — a defined contract that the chat could operate against. A short while later, that skill became a plugin.

But all of this happened because I wanted to write a particular spec. The plugin didn't exist before the session began; it emerged as part of the work of doing that first spec. And once the schema, the data layout, and the views were in place — the contract — the conversation simply continued. It segued from "let's build the methodology infrastructure" to "now let's use it to write the spec." Same chat, same session, same workspace. The right pane was now showing the spec being authored, with the chat alongside, both editing the spec and continuing to refine the methodology as new edge cases came up.

This is the pattern in concrete form. The plugin wasn't a prerequisite for the work — it was an artifact the work produced on its way to producing the actual deliverable. The chat fabricated the structured environment the work needed, exactly when the work was ready for it, then kept operating in that environment to finish the job. Schema, data, and views giving the LLM a place to put state. The file system holding it all. The chat staying present throughout, both shaping the application and operating it as the work evolved.

Why this matters

Most knowledge work has the same shape as this example. It starts unstructured and matures into something that needs structure. Today, the typical answer is: do the early work in chat (or on paper), then move to a separate tool when the work outgrows chat's limits.

But the move to a separate tool is costly. The continuity of the session is lost. The LLM that helped frame the work no longer has access to the context. The user has to rebuild the connection between thinking-about-the-work and operating-on-the-work in the new tool. Often the new tool isn't actually a great fit — it was built for a class of work, not for this specific journey, and the user spends as much energy bending the tool to their work as doing the work itself.

The chat-becomes-a-workspace pattern doesn't require this break. The schema, data, and views needed for the mature stage of the work can be fabricated during the session, by the chat, in response to what the work has become. The result is a custom-fit application — built for this specific journey, by the chat that was helping with it all along.

This is what the Living Workspace mode actually gives you, when it's used well. The previous article noted that B2 is barely emerging as a category. This pattern — chat becoming a workspace through a schema-data-views contract — is one of the most useful things B2 enables. Most products that nominally fit B2 today aren't doing it yet. The enabler is the file system underneath, with a contract on top. The plugin we built for spec writing is one proof of concept. There's a long list of knowledge journeys that could be supported the same way. Almost none of them are, today.
