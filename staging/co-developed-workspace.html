The Co-Developed Workspace

By Cliff Rosen, Founder & CTO, Orchestrator Studios

April 2026

A previous piece established that orchestration now lives in three places — a human head, a model's per-turn reasoning, and compiled software — and that the middle location is configured by files in the workspace the agent reads. This piece asks the practical question: what does an operator do with that fact?

The substrate accumulates

Start with an empty directory and an agent. The first time you do a piece of work — write a spec, run a literature scan, draft a memo — you carry the orchestration in your head. You issue instructions, the agent acts, you adjust. The work gets done. Nothing structural has happened. The directory holds the deliverable. The orchestration evaporates with the conversation.

Now do the same kind of work a second time. You notice yourself giving the same instructions you gave before. The smart move is to capture them. Drop in an example of the previous deliverable. Or write a short rules document — use this section order, never use bullet lists in the executive summary, cite at least two primary sources per claim. The next run starts with that material in scope. You issue fewer instructions, because the substrate is doing some of the work.

What just happened is mechanically obvious but conceptually significant. Orchestration that lived in your head on run one now lives in a file on run two. The same agent, the same tools, the same task — but the work goes faster, more consistently, with less attention from you.

This is the move that matters. Not "build a tool." Not "ship a workflow." Just: take orchestration out of your head, one piece at a time, and put it in the directory the agent reads. The substrate accumulates.

The ladder of encoding

There is a ladder of how much orchestration can be encoded into the substrate. The rungs are continuous, not discrete. Each is a single file added to the directory.

  1. Improvisation. Empty directory. The operator drives every step.
  2. Exemplar. A completed sample of the deliverable lives in the workspace. Structure and style transfer implicitly.
  3. Annotated exemplar. The sample teaches itself. Invariants and rules called out alongside the example.
  4. Rules document. A separate policy file governs how the agent operates, independent of any specific deliverable.
  5. State map. The workflow's shape is declared — the legal phases of the work and the transitions between them.
  6. Tools. Scripts and validators that perform or enforce specific transitions sit alongside the work.
  7. Skills. Reusable procedural know-how packaged so the agent can pull it in across sessions.
  8. Sub-agents. Specialized roles — drafter, reviewer, researcher — with their own instructions, invoked as needed.
  9. Plugin. Everything above, packaged for distribution.

There is no chasm between "doing it by hand" and "building a tool." The ladder is climbed one rung at a time when the work pattern justifies it, and stopped when it doesn't. A one-off task lives at rung one. A repeating task that matters lives somewhere higher. The economic question is which rung this work earns.

The climb is a conversation

What is striking about this ladder is that the operator doesn't have to step away to climb it. They don't open a different application, hire a developer, or write code. They climb in the same session in which the work is happening, by talking to the agent.

A typical climb looks like this. The operator is doing the work. They notice they keep correcting the agent on the same dimension. They say: whenever you draft a section, follow this rule. The agent says: want me to capture that as a rules entry in the workspace? They say yes. A file is written. The next turn — and every future turn — picks it up.

The conversation is no longer just about the deliverable. It is partly about the deliverable and partly about the methodology that produces it. The deliverable lands in the directory as content. The methodology lands in the directory as governance. Both are files. Both are subject to ongoing edit. The line between using the system and shaping the system has disappeared.

This is the real unlock. Not that operators can build applications — they always could, with enough time and money. The unlock is that operators can shape the orchestration of the work as a side effect of doing the work, by talking. The substrate accumulates while the work proceeds.

The asymmetry that determines what to encode

There is one more observation worth naming, because it determines what is worth investing in.

Of the two expertises behind any piece of knowledge work, only one is portable. Domain expertise — the judgment about what makes the deliverable good — is tacit, situated, and tied to the person who has it. It does not transfer well. A domain expert can produce great outputs, but encoding their domain expertise into files almost always loses something.

Orchestration expertise is different. The structure of the work, the decomposition, the sequence, the gates, the validators, the templates — all of this can be encoded with very little loss. A skilled orchestrator's playbook for a class of work can be written down, dropped into a directory, and run by an operator with no special background. The orchestrator's expertise transfers in a way the domain expert's does not.

This asymmetry has real economic consequences. Domain expertise stays with the people who have it. Orchestration expertise can be packaged, productized, and shared. Most of what is worth investing in encoding — for an individual operator, for a team, for a vendor — is on the orchestration side.

What to do

For an operator: every time you do a piece of work, ask whether any of the orchestration in your head wants to come out and live in the directory. If it does, the cost of moving it is one file. If it doesn't, leave it. The substrate should be exactly as thick as the work earns and no thicker.

For a team: the same, but coordinated. The substrate is shared. What one operator encodes, others inherit. The team's collective orchestration accumulates in the workspace itself, not in tribal knowledge.

For a studio: the work is to encode someone else's orchestration into someone else's substrate. That is the productizable layer. The operator brings the domain expertise. The studio brings the orchestration, packaged as files that an agent operates within. Neither party has to become the other. Both pieces of expertise stay where they belong.

Orchestration used to be either in a head or in shipped software. Now there is a third place — and it is the only one that updates in conversation. The work to do is figure out, for any given piece of knowledge work, what wants to live there.
