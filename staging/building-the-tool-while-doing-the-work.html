Building the Tool While Doing the Work

Every knowledge worker has always faced a tool-seeking question for the orchestration of their work. The Living Workspace adds a new option — build the tool while you're doing the work, fit to this specific project.

By Cliff Rosen, Founder & CTO, Orchestrator Studios

April 2026

The two expertises behind every piece of knowledge work

In an earlier piece, we drew a distinction that runs through every piece of substantial knowledge work: two expertises, both required.

  • Domain expertise — knowing the subject matter. What makes a good output good. What counts as a credible finding. What specific judgments matter for this kind of work.
  • Orchestration expertise — knowing how to break the work into manageable steps. How to sequence them. What quality gates to apply. What artifacts need to exist at each stage. How to know when something is done.

The domain expertise is usually the explicit thing — it's what you're hired for, trained on, recognized for. The orchestration expertise is often invisible. You only really see it when someone is missing it: when the work proceeds without structure, the artifacts don't add up, and quality holds at the level of whatever the person remembered to check.

The orchestration question every user faces

Orchestration expertise comes with a recurring question, asked silently for every task: how much do I lean on existing tools and process, and how much do I just go?

The question gets answered constantly, often without conscious thought:

  • If the task is running a CRM operation, of course you reach for Salesforce. The orchestration is largely embedded in the tool — pipeline stages, follow-up reminders, forecast rollups. The tool carries the structure.
  • If the task is writing an email, you don't reach for a tool at all. You start typing. The orchestration fits in your head, and tooling would just be overhead.
  • If the task is writing a proposal, you might start a conversation with chat — using it as a thinking partner and a place to draft. Light scaffolding, no formal structure, but more support than blank-page typing.
  • If the task is developing a financial model, you might pull up a template — someone else's orchestration, frozen in a spreadsheet, that you can adapt and fill in.

Whatever the task, the work is some sequence of steps. The user's judgment is about how much external scaffolding to lean on for that sequence: a full app, a template, a chat session, or just themselves.

The gap: when work needs orchestration but no tool fits

The interesting cases are when the work is substantial enough to need real orchestration support, but no tool exists for it specifically — or building one would cost far more than the project is worth.

A consultant putting together a board deck for a one-time client. A researcher running a multi-week analysis with custom requirements. A team scoping a small initiative with its own particular shape. A founder drafting a strategy memo that has to thread several specific stakeholders. Each of these has structure, stages, quality requirements — and no off-the-shelf tool that fits.

For work like this, the traditional options have been three:

  • Use a generic environment — Word, Google Docs, a notebook — and hold the orchestration in your head. The work proceeds, but the structure stays invisible and quality holds only at whatever the user remembers to check.
  • Cobble together generic tools — a doc, a spreadsheet, a Trello board, a few separate AI conversations. Stitch them together informally. Better than nothing, but the seams show, and the orchestration still mostly lives in the user's head as they navigate between tools.
  • Build a system — a custom app, a structured workflow, real software. Almost always too expensive for any project that won't be repeated many times.

Most ad-hoc knowledge work falls into the first two categories. Orchestration is invisible, quality is uneven, and the work tends to look messy at the edges — not because the people doing it lack orchestration expertise, but because there's nowhere good to put it.

The Living Workspace as a new option

The Living Workspace adds a new option to this question. You can build the tool while you're doing the work — and the tool is right-fit for this specific project, not a generic template that has to be bent into shape.

In a Living Workspace (Claude Code being the canonical example), the work and the orchestration infrastructure happen in the same chat session. You start working. As the shape of the work becomes clearer, you ask the chat to set up the orchestration: a folder structure for the artifacts, a schema for what counts as a complete unit at each stage, a custom view that shows progress, a quality check at the gates. The infrastructure emerges out of the work itself, fit to the project's specific needs.

This is materially different from the previous three options:

  • Better than holding orchestration in your head: the structure is externalized, persistent, reviewable. The user's orchestration expertise becomes legible — not just to themselves but to anyone who picks up the workspace.
  • Better than cobbling: the workspace is coherent. One place, one chat, one substrate, one set of artifacts that fit together — instead of a stitched-together set of tools the user has to navigate between.
  • Less expensive than building a system upfront: the cost of fabricating the tool folds into the cost of doing the work. There's no separate build phase, no multi-month payback hurdle. The tool exists by the end of the project because the project produced it.

What this changes for the orchestration question

For tasks that previously fell into the gap — work that needed orchestration but didn't justify a custom tool — the Living Workspace shifts the trade-off. Custom orchestration becomes feasible for any project, including a one-week one. The economics that kept tooling out of ad-hoc work are different now.

The deeper change is to what happens to the orchestration expertise itself. Previously, orchestration know-how mostly lived in the user's head — applied silently, never externalized, lost when the user moved on. With a Living Workspace, that know-how can take the form of artifacts in the workspace: schemas, views, processes, quality gates, captured as the work proceeds. Some of it becomes reusable for similar work down the road; much of it stays project-specific. Either way, the orchestration is now visible, examinable, and shapeable in collaboration with the chat.

This isn't an argument for replacing existing tools. CRMs still belong in Salesforce. Financial models still belong in spreadsheets. Email is still best handled by typing. But for the long tail of knowledge work that has no obvious tool home — and that's most of the work any individual contributor or small team actually does — there's now a real third option, beyond "hold it in your head" and "cobble it together." That's a meaningful change in a question every knowledge worker has been answering, often unconsciously, for as long as knowledge work has existed.
