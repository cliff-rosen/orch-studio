Six Modes of Chat and Substrate

By Cliff Rosen, Founder & CTO, Orchestrator Studios

April 2026

Apps before chat

Before chat became a productive capability, the structure of an application was settled territory. An app, in its general form, decomposes into three things:

  • The user — the person doing the work.
  • The substrate — the structured persistent state the app reads from and writes to. A CRM has records. A word processor has documents. A spreadsheet has cells. A file system has files and directories. The substrate is what gives the app its memory and its shape.
  • The UX — the interface for interacting with the substrate. Buttons, forms, lists, dashboards.

Not every part of an app is everyday-use. Many apps include fabricator portions — places where the substrate's structure itself gets shaped, distinct from the everyday-use portions where the user operates within an existing structure. Salesforce has Setup, where admins create custom objects, define fields, configure validation rules. Word has its template-and-style system, where document structure gets defined separately from filling that structure with content. Excel has macros and named ranges. The fabricator portion and the everyday-use portion are often separate experiences — sometimes for different roles (admin vs. end user), sometimes just for different mental modes.

These patterns were established decades before chat. The architecture of substrate + UX + user, with a fabricator/use distinction layered on top, is more or less how every productive piece of software was built and operated.

Then chat showed up

Three or four years ago, a new capability arrived in productive form: chat. Specifically, a user sending messages to a large language model running in a loop with tools — an LLM that can call tools, see their results, and continue reasoning across multiple turns within a single user prompt.

Even before LLMs made chat truly viable, the app ecosystem had been trying to incorporate chat-like interfaces, typically as support sidekicks. A help bot tucked into a corner. A question-answerer next to the real product. The pattern was known, even if the experience wasn't great. After ChatGPT in late 2022, this remained the obvious first move: bolt an actual capable chat onto a traditional app and let it explain the UX, answer questions, walk users through tasks. Many products shipped exactly this. A fine starting point — but a small one.

Where chat could go beyond the sidekick

The bigger question is what chat could do for app design and use beyond the sidekick role. A few possibilities surface as soon as you start asking:

  • Could chat be the entire app? Imagine the whole Amazon experience as a chat conversation. Some of it would work — you could ask the chat to search for products and get visual results back. But there are things you'd still want as fixed UI: a shopping cart in a specific place, always accessible, with consistent state; a stable navigation; an account view you can find without asking. Some affordances need to be persistent fixtures, not conversational outputs.
  • Could chat be how you take action, while the UI handles everything else? You'd still browse, scan lists, see your cart in its corner. But to actually do something — place the order, change the shipping address, apply a coupon — you'd just tell the chat. Chat as the action layer; the rest of the interface as the seeing-and-browsing layer.
  • Could chat build the app in the first place? Describe what you want and have a working application come out the other end. Vibecoding, in current form.
  • Could chat both build and use the app, in the same session, indefinitely? Could the app keep evolving as the work it supports evolves? Less obvious. Much rarer in practice.

Each of these is a different way chat could relate to apps. The table below names six modes that occupy this space, anchored on real products. Three or four years on, the field hasn't converged on which configurations work best for which purposes — products exist at every position, but the picture is unsettled. What follows is an organized look at the state of play.

The six modes

Here is the rubric in tabular form, then a walkthrough below. The example column names the host platform; the substrate column describes what that platform manages or operates on.

Mode Examples Chat builds the app? Chat operates the app? Substrate
A1 Classic App Salesforce, Word, Excel No (built by developers / admins) No Backend-managed state
A2 Assisted App Microsoft Office with Clippy No Helper only — chat explains the UI, doesn't move state Host app's backend state
A3 Chat-Driven App TableThat No (developer-defined schema) Yes — chat is the primary write interface; UI handles consumption Fixed schema (rows, columns, records)
A4 Pure Chat ChatGPT, Claude N/A — no app N/A — no app None. Conversation is a medium, not a substrate
B1 Vibecoding Platform Lovable, Bolt, Replit Agent Yes — chat builds the app No — chat exits when the build ends The app being built (becomes a Classic App after chat exits)
B2 Living Workspace Claude Code Yes Yes — same chat session, integrated with the build The file system itself

A1 — Classic App

Apps with no chat at all. The substrate (records, documents, spreadsheets) was shaped at design time by developers, with possibly a fabricator portion the user can access through the same UX (Salesforce Setup, Word's template editor). Everyday use happens through the UX. Salesforce, Word, and Excel are canonical examples. This is still the default for the vast majority of software in production today, and has been for decades.

A2 — Assisted App

An app whose UX is augmented with a helper chat. The chat is bolted onto the app to explain how to use the UX, point the user at features they missed, narrate what's happening. Crucially, it doesn't move state — it advises on use, but doesn't perform the using itself. Microsoft Office with Clippy, in the late 1990s, is the canonical (and famously unsuccessful) example. Note that Clippy itself is just the chat layer; the substrate it sits next to is Office's, not Clippy's. This mode is largely historical now — modern AI assistants in productive apps tend to actually move state, putting them somewhere else in the rubric.

A3 — Chat-Driven App

The substrate's schema is fixed up front by the app developer. Chat is the primary path for creating, updating, and transforming things within that schema. The UX next to the chat is the primary interface for consumption — viewing, browsing, reading. Manual UI as a "fallback" applies only on the write side. TableThat is the canonical example. The user can technically click through forms to add a row, but the intended path is to describe what they want and let the chat do it.

Even though chat is the primary write interface, the Chat-Driven App is still in Branch A — because the shape of the substrate (what a row is, what a column is, what a record means) is given by the app developer. The chat fills the schema; it doesn't author it.

A4 — Pure Chat

ChatGPT and Claude in their default configuration. There's no app of record beneath the conversation. Artifacts can emerge from chat — text, code, summaries, ideas — but they live in the chat thread, with no organizing principle beyond message order and "projects" as loose containers.

This is why nobody tries to use ChatGPT instead of a CRM. Not because the model can't reason about customer data, but because there's no substrate to read from, write to, or operate on with structure. The conversation is a medium where things happen, not a substrate that things live in. The mode is common as a standalone product category but sits outside the four-corner question because there's no app to fit chat relative to.

B1 — Vibecoding Platform

A platform whose chat is used to create an app or system, then exits. Lovable, Bolt, and Replit Agent are canonical examples — chat-driven environments whose purpose is to generate running applications from conversation. During a build session, the chat shapes a substrate that will become an app. Once creation is done, chat exits — or is replaced by a different chat that isn't the same conversation and isn't capable of continuing the substrate-generation discussion. Two temporally separate phases: build with chat, then use without it.

The artifact has to stand on its own once built. The output of vibecoding typically lives in Branch A territory afterward — most often a Classic App. In a real sense, vibecoding is a production process for Classic Apps. The chat is a temporary tool. The artifact must be coherent, finished, capable of operating without the conversation that produced it. This category has been expanding rapidly.

B2 — Living Workspace

A platform whose chat creates a substrate and stays alongside it throughout. Building, using, refining, extending — all happen with the same chat present. The substrate is the file system itself — files and folders, not a database-backed schema — which means its shape is open-ended and remains shapeable by chat as the work proceeds. Claude Code is the canonical example.

The fact that Claude Code's substrate is the file system is its defining property — and what distinguishes it from chat tools like Claude or ChatGPT (which have no substrate at all). The file system is an unusually general substrate: general enough that what's stored in it can describe, configure, or implement another app entirely (a CRM, a website, a data pipeline). Build something inside Claude Code and the artifact you produce can itself be substrate-shaped. While in Claude Code, though, only one substrate is being shaped: the file system. The artifact's eventual runtime substrate (its database, if it has one) only exists after deployment, at which point you've left the Living Workspace.

There's a second structural property worth naming. The file system is a low-level interface to the substrate. It's not where most of the user's interaction actually sits; it's the layer beneath. What Claude Code does is let the user choose — or even fabricate — the higher-level interface they want to operate the substrate through. Often that's a code editor, especially early in a session. But it can be anything: a terminal, a notebook, a Markdown viewer, a custom dashboard, even an HTML interface that only exists because the user asked Claude Code to make it during the session. The UX in the Living Workspace isn't fixed; it's selectable, and even fabricable, as part of the same chat that shapes the substrate.

This is the only mode where the chat that defined the substrate is still there when the substrate is being used, and where the using produces more shaping, indefinitely. Building and using are no longer separate phases; they happen in the same session, fluidly. The mode is barely emerging as a category — Claude Code sits nearly alone in the position.

Two surprising absences

Walking through the modes surfaces something the rubric makes visible: two of the corners are surprisingly empty given how long chat has been a productive capability.

The expected mainstream that didn't materialize: A3

When chat first arrived as a capability, the obvious bet was that most apps would adopt it as the primary way for users to manipulate their substrate. Why click through five forms when you could describe what you want? Why navigate menus when you could just ask "show me last month's revenue by region, broken down by product line"? You'd expect, three or four years in, that almost every existing productive app would have a chat-as-primary-write-interface configuration shipping or close to it.

In practice, this hasn't happened. Most apps are still A1 (no chat), or A1 with an A2 helper sidebar tacked on — chat that explains the UX rather than driving it. Genuine A3 products, where chat is the primary way to write to the substrate and the UX is for consumption, remain rare. TableThat is one of the few. The expected mainstream of chat-driven productive software didn't arrive on the timeline most people would have predicted.

The configuration barely on the conceptual map: B2

The Living Workspace is the configuration where chat is in both building and using, with the same chat session fluidly spanning both — building the substrate and operating it, integrated, persistent throughout. In traditional software, building an app and using it are separate phases — different people, different times, different contexts. B2 is the configuration where these phases blend.

Vibecoding (B1) is everywhere now; that's chat-in-building-only, with chat exiting before use. But B2 — chat in both, simultaneously — is barely emerging. Claude Code is the canonical example, and it sits almost alone. Most products treat building and using as separate phases (B1 then A1), inheriting the long-standing pre-chat assumption that the boundary between them matters. The idea that the same chat session can fluidly do both — and that the substrate can keep evolving as a result — is not yet a common reference point in how people think about chat-driven products.

What the absences add up to

Three or four years after chat became a productive capability, the corners you'd expect to be most populated and the corners you'd hope would be most explored are both still mostly empty. The configurations that have proliferated are A4 (pure chat as a standalone tool) and B1 (vibecoding). The configurations that would integrate chat with the everyday substance of productive work — A3 in established apps, B2 as a new shape — are surprisingly thin.

This isn't a thesis the article wants to defend. It's an observation the rubric surfaces by laying the modes out and asking which corners are inhabited. Why these particular configurations are where activity hasn't gathered yet — that's a question worth asking next, but in another piece.
