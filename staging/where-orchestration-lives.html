Where Orchestration Lives

By Cliff Rosen, Founder & CTO, Orchestrator Studios

April 2026

Knowledge work decomposes into atomic operations — read, search, evaluate, summarize, draft. We've written about this before. But naming the operations is only half the picture. The other half is orchestration: deciding which operations to perform, in what order, with what intermediate structure, against what success criteria.

Every piece of knowledge work draws on two distinct expertises that are easy to confuse. Domain expertise is what makes a good output good — the judgment about quality, relevance, accuracy, fit. Orchestration expertise is what makes the path to that output reliable — how to decompose, sequence, structure, evaluate. Both are always required. They are separable. They can come from different sources.

This piece is about the second one, and specifically about a fact that is easy to miss: orchestration always lives somewhere, and where it lives has changed.

Orchestration is locatable

Orchestration always lives somewhere. It never lives nowhere. In a single operator's head, executed turn by turn. In a written checklist taped to the wall. In a runbook in a wiki nobody reads. In a Salesforce instance with a hundred custom integrations. In a purpose-built application a vendor shipped twenty years ago. Asking where does the orchestration live for this work? is always a well-formed question.

Locatability surfaces a spectrum. On one end, orchestration is fully in a person's head, improvised in real time. On the other end, orchestration is fully encoded in compiled software, built by developers and shipped as a product. Everything in between — templates, runbooks, scripts, internal tools — is some degree of externalization, some amount of orchestration moved out of heads and into durable artifacts.

This spectrum has always existed. It predates LLMs. It is how knowledge work has always been organized. And until recently, it had two and only two endpoints.

Where agents fit

An agent is an LLM in a loop with tools and instructions. Given a task, it decides which tool to call, when the sub-task is done, whether to loop, what to do next. We've written elsewhere about agents as workers and planners. Here we want to be more precise about what kind of orchestration the planner mode actually performs.

It performs within-turn orchestration. The user issues an instruction. The agent then makes a series of decisions — tool choice, retrieval strategy, follow-up reasoning — until the turn concludes and control returns to the user. The user, between turns, decides what to attempt next. The agent, within the turn, decides how that attempt unfolds.

This is a real division of labor. It is also a new one. Before agents, orchestration happened either inside a human head, turn by turn, or inside compiled software, predetermined at build time. Within-turn model orchestration didn't exist as a category. It does now.

The third location

So orchestration can now live in three places:

  • In a person's head, expressed turn by turn.
  • In a model's per-turn reasoning, governed by whatever instructions and tools it has been given.
  • In compiled software, fixed at build time.

The middle location is new. And it is structurally different from both of its neighbors. It is faster than a human and more flexible than compiled software. It can be reshaped without a release. It is the first orchestration substrate that is malleable in the moment of use.

The crucial property of the middle location is what configures it. The agent's per-turn reasoning is not a black box. It is shaped by its system prompt, by the tools it can call, by the documents it has loaded, by any conventions present in its working environment. All of those are files. All of those can be edited. All of those can be authored by the same operator who is using the agent.

The substrate shapes the agent

In Claude Code and similar environments, this manifests directly. The workspace is a directory. The agent reads it. Some of the files in the directory are the work itself — drafts, data, notes. Some are tools the agent can invoke. And some are governance: a rules document, an annotated sample, a state machine declaring what transitions are legal, a skill describing how to approach a class of task.

Governance files don't run anywhere. They aren't compiled. They are read by the agent at the moment of need and shape its behavior on the spot. Edit one, and the agent's next turn behaves differently. Author a new one mid-session, and the workflow has changed.

What this means is that the orchestration living in the agent's per-turn reasoning is not fixed. It is whatever the substrate currently encodes. The same agent, in two different directories, will orchestrate two different ways. The same directory, after a small edit to its rules file, will produce different agent behavior on the next turn.

The new question

The old question — do I do this in my head, or do I get someone to build me software? — assumed orchestration had two homes. The new question is sharper: for this work, right now, how should orchestration be distributed across the three locations? How much in the operator's head? How much in the substrate the agent reads? How much in shipped software downstream?

And — because the substrate is just files, and the operator is the one writing them — that distribution can change continuously. There is no point at which orchestration is "decided." It is always being authored, somewhere, at some level of formality. The relocation matters because it makes orchestration something you tune rather than something you commission.
