§ 1 — Overview

The whole flow in one breath

The user types prose. One AI call turns that prose into structured JSON. The JSON is sprayed into five tables. Later, when the user asks for a memo, the pipeline reads those five tables back into plain text and feeds it to Claude fourteen times — once per section of a fixed document outline. The fourteen outputs, stitched together, are the memo.

Key mental model. The five tables are not inputs the user fills in. They are a normalized projection of a single AI analysis of the user's prose. Everything in the memo traces back to that one blob of text — or to economic data added by a separate pipeline outside Investigate.
§ 2 — Starting Point

What the user actually gives us

On the Investigate page, the user provides one blob of text. No forms, no structured fields, no dropdowns. Just prose.

Option A — Free text

"I suspect the major UK supermarkets are coordinating on own-brand pricing — margins look suspiciously identical across Tesco, Sainsbury's and Asda for the same goods."

Option B — Document upload

A .docx, .pdf, or .txt file. The page extracts the raw text in the browser before sending it to the backend. Optionally, the user supplies a title; if not, the filename is used.

And that's it

There is nothing else the user explicitly tells the system at this point. No defendants to name, no sector to pick, no evidence to list. The entire richness of the downstream memo will be inferred from whatever is in that one blob.

Why this matters for product. The quality of the final memo is bounded by the specificity of the input prose. A vague paragraph produces a memo with placeholder defendants and thin evidence; a detailed paragraph (or a full document) produces a memo with named parties, real numbers, and citable sources.
§ 3 — The Turning Point

The highlighted values are injected at call time from the user's input. Note the 10,000-character cap on the input text — anything longer is silently truncated before being sent to Claude. The model's response is parsed as JSON; if it comes back with markdown fences or extra text, the parser strips them.

This is the single moment where unstructured prose becomes structured data. Everything downstream — every table, every memo section — works from this JSON. There is no second AI call in Investigate that adds facts; there is only this one extraction, followed by the memo pipeline that reasons about those facts.

What happens next depends on confidence

The 0.25 gate. If Claude returns bayesian_posterior < 0.25, the investigation completes but produces no hypothesis, flag, or evidence rows. The user sees the analysis in the UI, but nothing enters the memo pipeline. This is a quality gate — low-signal ideas don't pollute the case queue.
§ 4 — The Case File

Five tables, all joined by one flag_id

The AI's JSON doesn't get stored as one blob. It's split across five tables — each a different view of the same analysis. This normalization is what lets the memo pipeline (and the review UI, and the dashboards) query the case different ways.

§ 5 — The Pipeline

14 stages, like a writer working through an outline

When the user asks for a memo, the pipeline doesn't make one big "generate memo" call. It makes fourteen, one per section. Think of a writer working through a fixed document outline, with the case file open on their desk, able to see everything they've already written.

What every stage sees

Each of the fourteen Claude calls receives exactly three inputs:

  • Case context — the five tables, stringified into one plain-text block. The same block every stage. Identical reference material.
  • Stage prompt — pulled from the prompt_library table. A generic, reusable instruction like "write Section VI: compute HHI, Lerner, and coefficient of variation." No case data in it.
  • Prior sections — a truncated trail (3,000 chars, most recent first) of what earlier stages wrote. Empty for stage 0, longest by stage 13.

7Sections XI–XII— defence analysis & certificationSONNET
Anticipates the defendants' counter-arguments (stress tests) and addresses class-representative viability (CPO).
8Sections XIII–XV— budget, risk, conclusionsSONNET
Investment Decision Matrix — budget, risk assessment, conclusions. The "should we take this case?" answer.
9Annexes A–C— glossary · calculations · contactsSONNET
Reference material: glossary of terms, supporting calculations, expert contact list.
10Annexes D–F— funders · financials · damages workpapersSONNET
Financial deep-dives supporting Sections IX–X.
11Annexes G–R— cases · timeline · PAEV · endnotesSONNET
Comparable-case write-ups, timeline, PAEV annex, endnotes.
12QA Certificate— QP-1 through QP-11 quality checksOPUS
An auditor pass over everything written so far. 11-point quality enforcement. Uses Opus because the cost of false approval is high. Must pass before Section I is written.
13Section I — Executive Summary— composed lastWRITTEN LASTSONNET
Critically, Section I is generated last — not first. You can only summarize numbers that have already been written. Generating the executive summary first would mean predicting your own conclusions. By running it after QA, the summary reflects verified, auditor-approved figures.

Two decisions worth flagging for product

Why Section I is last
The executive summary quotes specific figures. Writing it first would force the model to predict its own later numbers, creating inconsistencies. Running it last makes it a faithful mirror of what's already in the memo.
Why Opus only for stages 0 & 12
Sonnet 4.6 is 3–4× faster than Opus for comparable quality on legal analysis. Opus is reserved for the two stages where rigour matters most: pre-flight validation and final QA.

§ 7 — Where Info Comes From

What originates where

When you read a finished memo, different parts of it came from different sources. Understanding which parts came from where tells you which parts will be weakest when the inputs are thin.

What you see in the memoWhere it ultimately comes from
Named defendants, market name, alleged violation, damages band The user's original prose, extracted by the single AI call.
Confidence posture, recommended action The Bayesian posterior from the AI extraction.
Economic numbers — HHI, Lerner, CR4 Not from Investigate. Either null (renders as "REQUIRES CALCULATION") or populated by a separate enrichment pipeline.
Comparable prior cases & precedents Not from Investigate. The precedent_matches table must be filled in separately, otherwise the section is thin.
Evidence URLs, citations Not from Investigate. Added by enrichment or manual editing.
Legal structure, section ordering, argument scaffolding The stage prompts in prompt_library. Generic, reusable, case-independent. This is the v8.0 template.
Cross-section numerical consistency (HHI quoted the same way in Sections VI, IX, I) The prior-sections trail passed stage-to-stage. Each stage sees what the previous stages wrote.
Quality attestation (QA certificate) Stage 12 — Opus reviewing everything written by earlier stages.
Three sources, one merge, fourteen times. The memo's facts come from the user's prose (via AI extraction). The memo's structure comes from the prompt library. The memo's internal consistency comes from the stage-to-stage trail. Every section is a merge of those three.
§ 8 — For the product thinker

Gotchas & key design decisions

Things that shape the user experience and are worth knowing if you're thinking about how this product behaves at the edges.

1. The 0.25 posterior gate silently drops weak signals

If the AI assesses the user's input at a Bayesian posterior below 0.25, nothing is written to the memo pipeline. The UI shows a result, but no flag, no hypothesis, no evidence rows are created. The user might not realize their investigation was effectively a dead end.

2. An Investigate-only memo will have placeholder economics

Investigate doesn't compute HHI, Lerner, or CR4. It doesn't fetch precedents. A memo generated immediately after an investigation — before any enrichment step runs — will show "REQUIRES CALCULATION" in the statistical analysis section and a thin or empty precedents section.

3. Only the first hypothesis reaches the memo

If a case somehow ends up with multiple hypotheses on the same flag, the memo pipeline uses only hypotheses[0]. The rest are silently ignored. Relevant if a future workflow lets users add alternative hypotheses.

4. The user's raw input always survives as evidence

Whatever the user typed gets stored verbatim (first 500 chars) as a Tier-3 evidence item. So even if Claude fails to extract any other evidence, the memo prompt always has something concrete to work from.

5. Depth affects cost, not structure

QUICK runs one Claude call and produces a ~500-word triage memo. STANDARD and DEEP both run all 14 stages — DEEP just asks for longer sections. The structural skeleton is identical; DEEP is just a fuller fill-in.

6. Stages can time out silently mid-pipeline

Each stage call has a 110-second timeout. If one fails, the frontend retries up to 3 times with backoff. If a stage completed server-side but the HTTP response was lost, the frontend detects this via the pipeline log and continues. Still, users can see the modal stall on a specific stage — that's a UX signal worth surfacing.
