Meta-schema reference v0 draft

The meta-schema is invariant across workspaces. Domain contracts (kinds, fields, states) are workspace-specific; the meta-schema describes the skeleton every workspace shares.

Required documents

  • goal.md — Purpose · Deliverable · Audience · Timeframe · Scope (in/out) · Success criteria
  • workflow.md — Stages (named, ordered) · Items (what flows, refs contracts) · Transitions (with guards) · Stage invariants
  • process.md — Decisions · Pivots · Spine candidates

Required directories

  • contracts/ — One file per kind: schema + states + transitions + invariants. Machine-readable.
  • data/ — Subdir per kind, file per instance, conforming to its contract.
  • views/ — Custom views. Mostly empty; plugin renders defaults.
  • escapes/ — Objects that resist the contract, captured raw with a why-it-doesn't-fit note.

Required machine state

  • meta.json — workspace_id · created_at · last_modified · status · current_stage · domain_hint · meta_schema_version

Lifecycle

  • bootstrap: goal not yet specified, or workflow/contracts not yet drafted
  • operating: goal/workflow/contracts in place; work is happening
  • wrapping-up: deliverable being finalized
  • archived: work complete

Folder layout

workspace/
├── meta.json
├── goal.md
├── workflow.md
├── process.md
├── contracts/
│   └── <kind>.contract.json
├── data/
│   └── <kind-plural>/
│       └── <instance-id>.json
├── views/
└── escapes/

Cross-cutting

  • Spine — load-bearing patterns the model proposes for promotion to contract
  • Escapes — objects that don't fit; trigger contract evolution
  • Classification — every user message lands at one layer; propagate top-down from there
