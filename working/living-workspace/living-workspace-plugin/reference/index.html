Reference

Living Workspace

A Claude Code plugin for end-to-end knowledge work that needs structure but has no off-the-shelf tool. You chat in your terminal, a structured substrate fills in on disk, and a browser dashboard shows it live. The plumbing — contracts, validation, rendering — handles itself.

Overview

Why this exists

Chat apps are great for parts of knowledge work — answering a question, drafting a paragraph, exploring an idea. But for end-to-end knowledge work — producing a research synthesis, revising a document deliberately, maintaining a system that grows over time — chat alone has nowhere to put the work. There's no substrate. Inputs, works-in-progress, derivatives, outputs all live in your head or scattered across files you have to manage yourself. Real orchestration isn't possible.

Claude Code solves the substrate problem at the lowest level: it reads and writes your file system. That's enough — every artifact, every relationship, every state can live in files. But the file system is too primitive for most knowledge workers. You end up with chat plus a folder of opaque JSON and markdown. No views, no validation, no methodology — just the user and a gazillion files, with all the burden of figuring out what to do with them.

This plugin is the layer in between. It overlays Claude Code with:

  • A higher-level substrate — structured stores with contracts, validation, and references. Files mean something.
  • A live browser UI — a dashboard that shows your workspace as it grows. Read-only in v0.
  • Methodology blueprints — packaged orchestration patterns for typical knowledge-work shapes. Optional starting points that come with both substrate and method.
  • A bootstrap conversation — Claude walks you through the goal and structure. You don't start from a blank workspace.

The result is turnkey: you sit at your terminal, run claude, open a browser. You describe what you're working on. As you work, the substrate fills in and the browser updates live. Accessible the way you'd use a chat app — but with structure underneath that supports real start-to-finish knowledge work.

The system in one sentence The user chats with Claude in a terminal. Claude drives changes through a structured substrate of JSON files. The plugin renders the substrate live in a browser.

The pieces

  • Terminal — runs claude. This is where chat happens.
  • Browser — shows the dashboard. Read-only in v0.
  • Workspace folder — a directory on your disk holding all your work as structured files.
  • Plugin — a local program that watches the workspace, validates changes, renders the browser view, and ships the rules and templates.

How you actually use it

You never edit JSON directly. You don't configure widgets or build forms. You describe what you want; Claude builds and maintains the structure. The browser shows the current state. The terminal is where you act.

Core concepts

Substrate

The substrate is the structured layer of the workspace — the files and folders that hold your work in a way the system can read, validate, and render. Everything you create lives in the substrate. Everything the agent edits is in the substrate. Closing the browser doesn't lose anything; reopening it picks up exactly where you left off because the substrate is the source of truth.

The substrate isn't a single shape. It's a composition of stores — independent containers, each with its own structure. A workspace might have a Tasks table, a Memo document, and a Schedule calendar — all sitting alongside each other, each with its own rules.

/ ├── workspace/ ← the substrate │ ├── _contract.json workspace-level rules │ ├── state.json current lifecycle state │ ├── goal.json the workspace's purpose │ ├── manual.md universal operating manual │ ├── method.md blueprint methodology (when installed) │ ├── stores/ all your data, organized by store │ ├── pages/ saved page compositions │ └── escapes/ things that don't fit yet ├── log.md narrative log of decisions └── dashboard.html rendered output (the plugin emits this)

Anything narrative or rendered (log, dashboard) lives at the project root. Everything structured lives under workspace/.

Stores

A store is a named container of one specific shape. You add stores to the workspace as you work. Each store has a name, a type (one of five primitives), and a contract describing what data is allowed inside.

The five primitive types:

Table

Structured records as rows × columns. Optional state machine per row. Examples: tasks, contacts, sources, claims.

Document

Prose body with optional structured front-matter. Markdown. Examples: memo, notes, spec, draft.

Calendar

Time-scheduled events with start/end times, optional recurrence, conflict detection. Examples: schedule, deadlines, meetings.

Tree

Hierarchical nodes with single-parent constraint. Acyclic. Examples: outline, organization chart, concept hierarchy.

Graph

Nodes connected by directed, typed edges. Cycles allowed; multi-parent allowed. Examples: people network, decision graph, entity relationships.

Composition between stores

Stores can compose into each other in two ways:

  • A Table can have a column whose type is Document. Each row owns its own document body. Useful when each record needs rich prose alongside structured fields — e.g., a Sources table where each row has structured metadata (title, url, date) plus a long Notes document.
  • A Document can embed other stores by reference. The document body has tagged regions that the renderer fills with a live store at view time — e.g., a Memo document that embeds the Sources table inline at one section and the Schedule calendar at another.

How a store is stored

Each store is a folder under workspace/stores/. Inside the folder: a contract file describing the rules, plus the data files conforming to those rules.

workspace/stores/tasks/ ├── _contract.json declares: type=Table, row schema, states, transitions, invariants ├── task-001.json a row ├── task-001.notes.md sidecar file for a Document-typed column ├── task-002.json └── ...

Contracts

A contract is the rules of a store — what data is allowed, what shape it takes, what state machines apply, what invariants must hold. Contracts are machine-checkable: the plugin validates every change against the relevant contract. Bad data doesn't get written; bad contracts don't load.

Contracts exist at two levels:

  • Workspace contract (workspace/_contract.json) — declares the workspace's lifecycle, required stores, and cross-store rules.
  • Store contracts (workspace/stores/<name>/_contract.json) — declare each store's shape, state machine, references, and invariants.

The shape of a store contract depends on the store's primitive type. A Table contract has rows, states, and transitions. A Document contract has a front-matter schema and embed rules. A Graph contract has node and edge schemas plus an edge type catalog.

Why "contract" The word emphasizes that this is an enforced agreement — between you, Claude, and the plugin. Claude can't write data that violates the contract. You can change the contract, but doing so is recognized as a structural change (not a casual edit).

Invariants

Beyond shape (handled by JSON Schema), contracts can declare conditional rules — "if state == done, completed_at must be set," for example. These are written in JSON Schema's if/then/else form so the plugin can enforce them automatically.

Widgets

A widget is a generic, reusable component that knows how to render a specific shape of data. The plugin ships a catalog of widgets; you don't author them.

The v0 widget catalog:

The same store can be rendered by multiple widgets — a Table of Tasks can appear as a smart-table on one page, a kanban on another, a calendar on a third (if it has a due-date field). Widgets compose; primitives stay the same.

Pages

A page is a composition of widgets. Pages are how you organize what you see. Each page descriptor declares: a title, a series of rows, and which widgets render in each row against which stores.

The dashboard is just one page — the default landing. A workspace can have many pages, each with its own composition.

A page descriptor (lives in workspace/pages/<name>.json):

{
  "name": "dashboard",
  "title": "Dashboard",
  "rows": [
    {
      "widgets": [
        { "widget": "smart-table", "store": "stores/tasks",
          "config": { "filters": { "state": ["next", "doing"] } } }
      ]
    },
    {
      "widgets": [
        { "widget": "kanban", "store": "stores/tasks" },
        { "widget": "list", "store": "stores/projects" }
      ]
    }
  ]
}

Multi-widget rows render side-by-side. Each widget references exactly one store. The plugin validates compatibility — if you ask for a kanban of a Table without a state field, the widget shows an error tile rather than rendering broken.

Page chrome vs page body

The dashboard composes from two layers:

  • Chrome — automatic, lifecycle-aware. Shell bar, phase strip, goal banner, "what's happening" callouts. Plugin-driven.
  • Body — your page composition. Workspace-driven, per page.

Orchestration

The substrate is the storage. Orchestration is the work pattern — the methodology that turns inputs into outputs through a sequence of structured stages.

A typical orchestration looks like:

  1. You start with inputs — raw documents, source materials, ideas.
  2. You extract — pull out what matters into structured records.
  3. You distill — boil down to the essentials.
  4. You structure — organize and connect.
  5. You compose — produce the output, drawing on everything above.

Each stage might use different stores — inputs in a Documents store, extractions in a Table, distillations in another Table or Tree, output in a Document. The orchestration is the flow between them.

Why orchestration matters Without an orchestration pattern, the substrate is just storage with no shape of work. Methodology gives storage a direction. Adding a "key-points" Table is structure; knowing when to extract key-points from highlights and how to use them in the draft is orchestration.

Orchestrations come baked into blueprints.

Blueprints

A blueprint is a packaged methodology + the substrate that supports it. Blueprints turn typical knowledge-work patterns into one-click starting points. When you install a blueprint, your workspace fills with the right stores, the right initial pages, and a methodology document the agent reads.

A blueprint contains:

  • README — the methodology in human terms. What this is for, when to use it.
  • method.md — the orchestration pattern. The agent reads this to make method-aware suggestions and stay consistent with the methodology.
  • Workspace contract — the lifecycle and required stores for this kind of work.
  • Initial stores — the kinds you start with, with their contracts pre-set.
  • Initial pages — the dashboards that surface progress.

Blueprints are optional. You can start blank. But for typical knowledge work, picking a blueprint saves a long bootstrap conversation and gives you a methodology that's worked elsewhere.

How blueprints surface

  • To you — during bootstrap, after your goal is defined, the dashboard shows blueprint candidates with their methodology summary. Even if you don't pick one, browsing them helps you see how others have structured similar work.
  • To Claude — once installed, method.md is part of the workspace's operating context. Claude consults it to suggest the next step in the methodology, push back on out-of-method moves, and surface stage transitions.

How it works

The two-window experience

You have one input channel (the terminal, where you type to Claude) and one output channel (the browser, where you see the workspace). They're separate windows but a single experience.

  • Terminal — Claude is here. You describe what you want; Claude does it. The terminal is where you write the workspace.
  • Browser — the dashboard is here. As Claude writes files, the browser updates live. You watch the workspace come into being. (Read-only in v0.)

You don't manage state. The system does. Closing either window doesn't lose anything; the substrate is on disk.

The plugin's role

The plugin is a small local program. It does four things:

  1. Bootstrap. A slash command starts a new workspace from the universal skeleton or a blueprint.
  2. Validate. Every file change is checked against the relevant contract before it's committed.
  3. Render. The workspace state is rendered to a live HTML dashboard, updated as files change.
  4. Watch. The plugin watches the workspace folder for file changes and pushes updates to the browser.

The plugin understands no domain. It doesn't know what a CRM is or what a competitive analysis looks like. All domain knowledge — what kinds you have, how they relate, what views you want — comes from the workspace itself, fabricated by Claude during use.

Lifecycle

Every workspace moves through a small set of named states. The default lifecycle:

goal-defining → contracts-drafting → operating → archived
  • goal-defining — you've just scaffolded a workspace. The agent walks you through articulating the goal: purpose, deliverable, audience, success criteria. Until the goal is filled in, no contracts or data can be added.
  • contracts-drafting — with goal in hand, you choose a blueprint (or start blank). Contracts get drafted; the agent customizes them based on your goal.
  • operating — the long phase. You're doing the actual work. Capture, extract, distill, restructure. Most of a workspace's life is here.
  • archived — terminal state. Read-only record.

The lifecycle is declared in the workspace contract; workspaces can extend with custom states (e.g., a single-deliverable workspace adding a wrapping-up state before archived). Backward transitions are explicit and allowed: you can pivot from contracts-drafting back to goal-defining if the goal needs revision, or from operating back to contracts-drafting if the rules need to change.

Operating loop

Once your workspace is in the operating state, the rhythm is the same on every user message:

  1. You say something.
  2. Claude understands the request.
  3. Claude considers: would satisfying this change the rules?
    • No — Claude acts. Validates the change. Renders.
    • Yes — Claude pauses, surfaces what would change, asks for confirmation, then acts.
  4. If a structural change happened, Claude logs it to log.md.

That's the entire operating discipline. Most user messages don't change rules — they're queries, data operations, view tweaks. The agent just acts. The structural-change gate is the one moment of explicit ceremony, and it's there because rule changes have downstream consequences.

Why the gate Adding a column to a Table contract means existing rows might not have that column. Changing a state machine might invalidate existing rows. Editing the goal might make the whole workspace's structure suspect. These need a deliberate moment, not an accidental drift.

Typical use cases

What kinds of work fit a Living Workspace? Anything that needs structure but doesn't have an off-the-shelf tool. Examples:

Document revision

You have an input document and want to revise it deliberately — not just edit in place, but extract what matters, decide what to change, write the revised output with intention.

Orchestration: inputs → highlights → key-points → changes → tone → output.

stores: inputs (Document), highlights (Table), key-points (Table), changes (Table), tone (Document), output (Document)

Research synthesis

You're reading a body of sources and need to produce a synthesized memo. The workflow is too involved for ad-hoc note-taking but doesn't justify a dedicated app.

Orchestration: sources → claims → themes → outline → memo.

stores: sources (Table), claims (Table, refs source), themes (Tree), outline (Tree), memo (Document, embeds claims)

Personal task management

A trustworthy single source of truth for everything you need to do. Captures fast, surfaces what's active, defers what's not ready.

Orchestration: capture → process → engage → review.

stores: tasks (Table), projects (Table), areas (Table)

Decision log

Make consequential decisions deliberately and keep a record of why. The workflow surfaces options, criteria, and rationale rather than just an outcome.

Orchestration: decision → options → criteria → analysis → resolution.

stores: decisions (Table), options (Table, refs decision), criteria (Table), analyses (Document store)

Competitive landscape

Survey competitors and produce a memo for a strategy meeting. Citation-disciplined; structured comparison.

Orchestration: competitors → features → comparison matrix → insights → memo.

stores: competitors (Table), features (Table), comparison (Table, refs both), insights (Table), memo (Document)

Reading review

Capture quotes, notes, and insights from books or papers in a way that compounds over time.

Orchestration: book → quotes → notes → insights → summary.

stores: books (Table), quotes (Table, refs book), notes (Document per book or shared), insights (Table)

These aren't built into the system. They're documented patterns that map to specific blueprints. Picking a blueprint installs the substrate plus the methodology. Skipping the blueprint and starting blank is also fine — the system supports both.

Working patterns

Capture is fast

When you're in capture mode — dropping new tasks, sources, ideas — the agent shouldn't interrogate. It accepts the capture, lands the record at its initial state, and moves on. Clarification happens later, in process mode.

Process happens deliberately

Periodically (or when prompted), you go through what's been captured and clarify each item — set its state, attach it to a project, decide whether to do it. This is process mode. The agent's job here is to surface what's pending and ask focused questions.

Engage moves things forward

Most of operating life is engaging — actually working items: marking them done, advancing states, capturing notes inline. The agent supports the work; you do it.

Review surfaces patterns

Cadenced review (weekly, monthly) brings out:

  • Stuck items — things that have been in the same state for too long.
  • Spine candidates — patterns that recur enough to warrant promotion to a contract field.
  • Escapes — things that don't fit the contracts and need to be resolved (force-fit is wrong; evolve the contract or leave as escape).

Structural changes are explicit

The agent doesn't quietly evolve contracts. When you ask for something that would change the rules, the agent surfaces what would change and confirms before acting. This keeps the substrate trustworthy across long sessions.

Glossary

Blueprint
A packaged methodology plus the substrate that supports it. Optional starting points for typical knowledge work.
Contract
The rules of a store (or the workspace as a whole). Machine-checkable. Lives in _contract.json.
Embed
Inline reference from a Document to another store. Renders the embedded store at view time.
Escape
A piece of data that doesn't fit any current contract. Captured raw with a why-not note instead of force-fit.
Goal
The workspace's anchor — why it exists. Captured in goal.json. Reread when orientation drifts.
Manual
The universal operating discipline. Lives in workspace/manual.md. Same content for every workspace.
Method
The blueprint-specific orchestration pattern. Lives in workspace/method.md when a blueprint is installed.
Orchestration
The flow of work — inputs to derivatives to outputs. The methodology that gives the substrate direction.
Page
A composition of widgets that renders some slice of the workspace. The dashboard is the default page.
Plugin
The local program that bootstraps, validates, and renders the workspace. Watches files; pushes updates to the browser.
Primitive
One of the five basic store types: Table, Document, Calendar, Tree, Graph.
Sidecar
A file alongside the main data file for a row, holding extra content. Used for Document-typed columns: task-001.notes.md alongside task-001.json.
Spine
Load-bearing patterns in the substrate. Recurring fields, frequent relationships — candidates for promotion to formal contract.
State (workspace)
The workspace's current runtime status. Lives in state.json. Conforms to the lifecycle in the workspace contract.
Store
A named container of one specific shape (Table, Document, Calendar, Tree, or Graph). Holds data plus its contract.
Substrate
The structured layer of the workspace — the files and folders that hold work in a way the system can read and validate. Source of truth.
Widget
A reusable component that renders a specific shape of data. Plugin-shipped; not user-authored.
Workspace
One project, one folder. Contains the substrate plus rendered output and narrative log.
